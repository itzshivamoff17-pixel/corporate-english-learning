How to Communicate During Technical Interviews

What to say when you're stuck, how to think out loud, and more.

January 18, 2026 · 9 min read

You can write perfect code and still fail a technical interview. Why? Because communication matters more than correctness.

According to a study by Triplebyte (2019), candidates who "think out loud" are 2.3x more likely to pass technical interviews, even when their solution isn't optimal. Interviewers aren't just evaluating your code—they're evaluating whether they want to work with you.

The "Thinking Out Loud" Framework

Most candidates go silent when coding. This is a mistake. Silence makes the interviewer nervous. They can't tell if you're:

  • Making progress
  • Stuck on a bug
  • Completely lost

Instead, narrate your thought process like a sports commentator.

The 5-Step Communication Flow

  1. Clarify the problem (2 minutes)
  2. Discuss the approach (3 minutes)
  3. Code while narrating (15-20 minutes)
  4. Test your solution (3 minutes)
  5. Optimize and discuss trade-offs (5 minutes)

Step 1: Clarify the Problem (2 minutes)

Never start coding immediately. Ask clarifying questions to show you think before you code.

Example Questions:

• "Can I assume the input array is sorted?"
• "What should I return if the input is empty?"
• "Are there any constraints on time or space complexity?"
• "Can I use additional data structures, or should this be in-place?"
• "What's the expected size of the input? Are we optimizing for small inputs or large ones?"

Why this works: It shows you're thorough and won't make assumptions in production code.

Step 2: Discuss the Approach (3 minutes)

Before writing any code, explain your high-level approach. This gives the interviewer a chance to course-correct if you're going down the wrong path.

Template:

"Okay, so my initial thought is to use a [data structure] because [reason]. The algorithm would be:
1. [Step 1]
2. [Step 2]
3. [Step 3]

This would give us O(n) time complexity and O(n) space. Does that sound reasonable, or would you like me to explore a different approach?"

Pro tip: If the interviewer says "That works, but can you do better?", they're giving you a hint. Don't get defensive—say "Great, let me think about optimizations."

Step 3: Code While Narrating (15-20 minutes)

This is where most candidates go silent. Don't. Narrate what you're doing as you code.

What to Say While Coding

When starting a new section:

"Now I'm going to initialize a hash map to store the frequencies..."

When writing a loop:

"I'm iterating through the array to find all pairs that sum to the target..."

When making a design decision:

"I'm using a set here instead of an array because we need O(1) lookup time..."

When you spot a potential bug:

"Wait, I need to handle the edge case where the array is empty. Let me add that check..."

What to Say When You're Stuck

This is the moment that separates good candidates from great ones. When you're stuck, don't panic in silence.

The "Stuck" Script

"I'm hitting a roadblock with [specific issue]. Let me think through this out loud..."

"One approach could be [Option A], but that would be O(n²). Another option is [Option B], which is more complex but faster. What do you think—should I prioritize simplicity or performance here?"

Why this works: You're showing problem-solving skills and inviting collaboration. Interviewers often give hints when you do this.

Step 4: Test Your Solution (3 minutes)

Don't wait for the interviewer to ask. Proactively test your code.

Testing Script:

"Let me walk through this with a test case. If the input is [1, 2, 3, 4] and the target is 5...
• First iteration: i=0, we check if 5-1=4 exists in the set. It doesn't, so we add 1.
• Second iteration: i=1, we check if 5-2=3 exists. It doesn't, so we add 2.
• Third iteration: i=2, we check if 5-3=2 exists. Yes! So we return [1, 2]."

Also test edge cases:

  • Empty input
  • Single element
  • Duplicates
  • Negative numbers (if applicable)

Step 5: Optimize and Discuss Trade-offs (5 minutes)

Even if your solution works, discuss how it could be improved.

Optimization Script:

"This solution is O(n) time and O(n) space. If space is a concern, we could sort the array first and use two pointers, which would be O(n log n) time but O(1) space. The trade-off is that sorting changes the original array, which might not be acceptable depending on the requirements."

Common Communication Mistakes

  • Going silent for 5+ minutes: The interviewer thinks you're stuck.
  • Saying "This is easy": Comes across as arrogant.
  • Apologizing excessively: "Sorry, I'm not sure..." shows lack of confidence.
  • Arguing with the interviewer: If they suggest a different approach, explore it.

The "Collaborative" Mindset

Treat the interview like a pair programming session, not an interrogation. Use phrases like:

  • ✓ "What do you think about this approach?"
  • ✓ "Does this align with what you had in mind?"
  • ✓ "I'm considering two options here. Which would you prioritize?"
  • ✓ "Is there a specific constraint I should optimize for?"
