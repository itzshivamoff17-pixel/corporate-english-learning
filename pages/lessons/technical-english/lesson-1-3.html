Architecture Decision Records - Lesson 3

Documenting Trade-offs & Decisions

"There are no solutions, only trade-offs." This quote by economist Thomas Sowell is the unofficial motto of senior engineering. This lesson teaches you how to document the why behind your code using the industry-standard ADR format.

The "No Silver Bullet" Principle

Fred Brooks, in his Turing Award-winning paper, argued that there is no single technology or methodology that causes a 10x improvement in productivity. Every choice has a cost.

The "Buy vs. Build" Dilemma

One of the most common trade-off discussions. Use this framework to articulate your choice:

Choosing "Buy" (SaaS)

"We are choosing Stripe over building a custom payments gateway.

Trade-off: We accept vendor lock-in and fees at the expense of lower control, to gain speed to market and reduced security liability."

Choosing "Build" (Custom)

"We are building a custom recommendation engine.

Trade-off: We accept higher maintenance costs and dev time at the expense of speed, to gain a proprietary competitive advantage that off-the-shelf tools can't provide."

The ADR Standard (ThoughtWorks)

ThoughtWorks Technology Radar and GitHub recommend using Architecture Decision Records (ADRs) to capture the context of decisions. This prevents the "Why did they build it this way?" confusion 6 months later.

Anatomy of a Perfect ADR

Michael Nygard proposed this widely adopted 5-part structure:

1. Title

ADR-001: Use PostgreSQL for User Data

2. Context

We need a reliable, ACID-compliant database for user transactions. We anticipate 10k users in Year 1.

3. Decision

We will use PostgreSQL 15 hosted on AWS RDS.

4. Consequences (The Trade-offs)

(+) Strong consistency and easy hiring.
(-) Harder to scale horizontally compared to DynamoDB.

5. Status

ACCEPTED

Reference: Nygard, M. (2011). "Documenting Architecture Decisions." thinkrelevance.com.

Phrasing Trade-offs in Code Reviews

When suggesting a change in a PR, frame it as a trade-off discussion rather than a command. This increases psychological safety.

The "Consider X vs. Y" Pattern

"I see you chose to use an in-memory cache here.

Observation: While this is super fast (low latency), it means we might lose data if the service crashes (low durability).

Suggestion: Have we considered using Redis so the data persists across restarts? It adds a network hop (latency), but improves reliability."

Common Trade-off Keywords

  • Latency vs. Throughput
  • Cost vs. Performance
  • Flexibility vs. Stability
  • Security vs. UX
